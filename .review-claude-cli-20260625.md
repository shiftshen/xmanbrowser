# Xbrowser 二次独立审核 — claude-cli (2026-06-25)

复核对象: main @ 25ef284。codex-cli 一审已跑通测试(31 passed / verify_stability ALL GREEN / 双引擎 smoke / proxy TH)。
本审重点挑刺，按严重度排序。**合规边界:干净** —— 全仓未见注册/支付/OTP/验证码/风控绕过代码;
proxy_providers.py 明确排除 register/payment。

## HIGH-1 控制 API 无鉴权 + CORS `*` → 任意网站可读代理凭据并控制 profile
- service.py:37-42 `allow_origins=["*"]`,全程无 token/依赖鉴权(grep 仅 proxy 的 has_auth)。
- `GET /api/proxies` (store._proxy_row) 原样返回 `raw`,内含 `user:pass`。
- 后果:用户任意浏览器访问的恶意页面可 `fetch('http://127.0.0.1:8723/api/proxies')` 跨域读到全部代理明文凭据;
  并能 POST 创建/删除/launch/stop profile、stop-all。绑定 127.0.0.1 挡不住浏览器侧 CSRF/读取(返回头 ACAO:*)。
- 建议:加本地 loopback token(启动写入用户目录,UI 读后带 header),CORS 收敛到 tauri origin;至少把 raw 凭据移出列表响应。

## HIGH-2 chromium 引擎 profile 之间 WebGL/canvas/audio 未隔离 → 同机可交叉关联
- _generate_chromium (fingerprint.py:131-159) 只存 UA/platform/screen/hw,**无 webgl/canvas/font** 噪声;
  summary 自承 "Chromium (real GPU)"。launcher 的 chromium 分支 (launcher.py:146-155) 仅应用
  user_agent/locale/viewport/screen/color_scheme。
- 后果:同一台机器上多个 chromium profile 暴露**相同真实 GPU** 的 WebGL vendor/renderer 与相同 canvas/audio hash,
  彼此可被指纹关联 —— 与"每 profile 唯一指纹"的卖点相悖(camoufox 引擎才有 per-profile 噪声)。
- 附带:存进 config 的 `platform / hardwareConcurrency / deviceMemory / languages` 在 launch 时**根本没被应用**
  (kw 里没有这几项),summary 展示的值是假象,实际暴露真机值。
- 建议:文档明确"chromium 引擎=干净自动化 stealth,但不保证同机跨 profile 的 WebGL/canvas 唯一性";
  若要隔离需注入 per-profile WebGL/canvas spoof 脚本。这是比一审候选更重的指纹缺陷,需老板知悉定位。

## MED-1 /api/batch/launch 绕过引擎检测(一审已发现)+ 并发首下竞态
- service.py:211-220 直接 `manager.launch`,无 single-launch 的 `engine.is_installed` 202 守卫 (对比 189-192)。
- 引擎缺失时批量会 fork N 个 runner:camoufox runner 路径(runner→_launch_camoufox)**根本没有 ensure 下载**,会静默失败
  (runner stdout/stderr=DEVNULL),profile 短暂显示 running 后被 reap。chromium runner 各自 `_ensure_chromium`,
  且与 engine.ensure_async 是**两条下载路径**,N 个 runner 并发写同一 ms-playwright 缓存 → 下载竞态/损坏风险。
- 建议:batch_launch 复用 launch_profile 的 is_installed 守卫,缺失则返回 202+progress 且只触发一次下载。

## MED-2 代理 auto-disable 不会自动恢复 + rotation 死代码
- record_proxy_check (store.py:466-471) 成功分支只清 fail_count,**不把 enabled 置回 1**;
  3 连败 auto-disable 后,即便 check_all_proxies 复查成功也永久停用(需手动开)。语义不对称,易让恢复的代理被埋。
- next_enabled_proxy (store.py:490-497) 轮转逻辑**无人调用**;profile 启动用固定 proxy_raw,所谓"rotation"未接线。
- 建议:成功时 `enabled=1`(或显式注释为人工恢复);明确 rotation 是否要接入。

## MED-3 engine.is_installed("chromium") 每次起 patchright driver → status 轮询昂贵
- engine.py:65-75 每次检测都 `with sync_playwright() as p` 启动 node driver 子进程;status()/status_all()
  被 UI 下载进度模态反复轮询 (App.tsx:813),每次 GET /api/engine/status 都 spawn 一次 driver,开销大。
- 建议:缓存可执行路径探测结果,或直接 os.path 探测 chromium 可执行文件而不起 driver。

- lib.rs 退出仅 kill sidecar;detach 出去的浏览器进程(posix setsid / win new-group)在 App 关闭后不被 stop_all,残留(或为有意)。

## 结论
一审 3 个候选属实(batch 守卫、CLI 无 engine、adhoc 签名)。更严重的两处遗漏:
(1) 本地 API 无鉴权+CORS* 导致代理明文凭据可被任意网页读取/控制(HIGH);
(2) chromium 引擎跨 profile WebGL/canvas 不隔离且部分 navigator 字段未真正应用(HIGH)。
建议优先修 HIGH-1/HIGH-2 与 MED-1,再处理其余。合规边界无问题。
