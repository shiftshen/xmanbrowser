# XmanBrowser v1.3.4 Stability Design

## Goal

Keep Camoufox and Chromium profiles responsive for long-running sessions, automatically recover a runner whose browser control channel has stalled, and keep profile cards usable with long names and notes.

## Root cause and approach

The runner currently sleeps with `time.sleep()` while it owns a synchronous Playwright connection. That does not pump Playwright transport events. The live v1.3.3 installation showed the Camoufox child as a zombie while the runner and Node driver remained alive for more than a day, so `running.json` kept reporting a browser that no longer existed.

The runner will use a short Playwright wait on an available page instead of a process sleep. This pumps transport events, retries another tab when one closes, and detects a closed browser. A per-launch random token binds the persisted record, heartbeat, and real process command together. An explicit `starting` heartbeat gets a 90-second allowance covering browser launch and the bounded 45-second navigation; the runner then switches to `running`, pumps every 400 ms, and writes a heartbeat every two seconds. The manager reaps a stalled current-version runner only after verifying that the PID command still carries the matching token. Process identity has match/mismatch/unknown states, and an unknown probe preserves state for retry instead of orphaning a live browser. Legacy runners without a token or heartbeat retain the previous PID-only behavior.

The profile grid will constrain every flexible child with `min-width: 0`, truncate long profile names on one line, and clamp notes to three wrapped lines. The full note remains available through the native title tooltip. Responsive toolbar, sidebar, spacing, and card breakpoints keep the main information and actions visible from 1280 down to 720 pixels without horizontal page scrolling. The desktop window now defaults to 1024×680 and permits resizing down to 720×520 instead of forcing 1200×820 with a 900-pixel minimum.

## Error handling

- A normal user window close is logged as a clean stop, not `LAUNCH FAILED`.
- A browser/driver stall stops heartbeats; the next status poll removes and terminates the stale process group.
- Heartbeat files are PID-and-token scoped so a stale file or reused PID cannot invalidate or terminate an unrelated process.
- A temporary `ps`/PowerShell/CIM probe failure preserves the running record and heartbeat so the browser remains manageable on the next poll.
- No profile, proxy, fingerprint, or user-data schema changes are introduced.

## Acceptance

- Unit tests prove the runner pumps Playwright events, survives one closed tab, throttles heartbeat writes, and never kills a mismatched/reused PID.
- A UI contract test proves the long-content containment rules remain present.
- Python tests, frontend build, Rust checks, and a real desktop/browser smoke pass.
- Desktop and narrow-width screenshots show no horizontal overflow and notes use at most three lines.
